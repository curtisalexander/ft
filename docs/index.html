1. What fine-tuning actually is

A modern language model is just a very large pile of numbers — its weights (also called parameters). Those numbers were set during pre-training, where the model read a huge amount of generic text and learned the statistical shape of language: grammar, facts, reasoning patterns, common formats.

Fine-tuning means taking that already-trained model and continuing to train it a little more on your data, for your task. You are not building a model from scratch — you are nudging an existing one. Concretely, you show it examples (input → desired output), measure how wrong it is (the loss), and use gradient descent to adjust the weights so it is a little less wrong next time. Repeat over your dataset a few times (a few epochs) and the model specializes.

The one-sentence version Pre-training teaches a model language in general; fine-tuning teaches it your task in particular by adjusting weights that already exist.

Because you start from a capable model, fine-tuning needs far less data and compute than pre-training — thousands of examples instead of billions, and a laptop or a single GPU instead of a data center.

2. The spectrum: prompt → RAG → fine-tune

Fine-tuning is not always the right tool. There is a ladder of effort, and you should climb it only as far as you need to:

TechniqueWhat it doesChanges weights?Reach for it when…
PromptingYou write better instructions / examples in the request itself (few-shot).NoThe base model already can do the task; you just need to ask well.
RAG retrievalYou fetch relevant documents and paste them into the prompt at query time.NoThe task needs knowledge the model lacks (your docs, fresh facts).
Fine-tuningYou permanently adjust the weights on labeled examples.YesYou need a consistent behavior / format / skill, lower latency, or a small specialized model.
Rule of thumb Fine-tune for behavior and form ("always output JSON like this", "tag phone numbers"); use RAG for knowledge and facts ("answer from these 10,000 documents"). Fine-tuning is a poor way to inject facts and a great way to lock in a skill or output shape.

3. Why some models can be fine-tuned and others can't

When Hugging Face says a model is "fine-tunable," it really comes down to three things being true at once:

  • The weights are published. You can download the actual parameter file (often .safetensors). To change weights, you must have the weights. This is what "open-weight" means.
  • The license permits it. Open weights still come with a license (Apache-2.0, MIT, Llama Community License, etc.). Most permit fine-tuning; a few restrict commercial use. Always read it.
  • The architecture is trainable on your hardware. A 7-billion-parameter model is "fine-tunable" in principle, but full fine-tuning may need more GPU memory than you have. (Techniques like LoRA — see §6 — fix this.)

So when you see "fine-tunable" on a model card, read it as: "the weights are downloadable and the license lets you keep training them." It does not mean someone has set up a special service for you — it means the door is open because the model is open.

4. Closed frontier models (ChatGPT, Claude, Gemini)

The core constraint You cannot fine-tune a closed model in the classic sense because you do not have its weights. The weights of GPT-4-class and Claude-class models live only on the provider's servers. No weights → nothing to adjust → no fine-tuning.

There are two important nuances, though:

  • Hosted fine-tuning APIs. Some closed providers (e.g. OpenAI for certain models) let you upload examples and they fine-tune a private copy on their infrastructure. You still never see the weights — you send data in, you get an API endpoint out. It's real fine-tuning, just rented.
  • Building on top of the model. Even with a fully closed model you can still specialize behavior without touching weights: rich system prompts, few-shot examples, RAG, tool use, and output validators. You're wrapping the model, not retraining it.

By contrast, open-weight models (Llama, Mistral, Gemma, Qwen, BERT, T5, DistilBERT, and thousands more on Hugging Face) hand you the parameter file. You can fine-tune them on your own machine, inspect them, and ship the result yourself. That freedom is exactly why this guide's example uses an open model.

5. Three model shapes — and what each is good at

"Language model" covers three architectures. Knowing which is which tells you what task to fine-tune for. They are all built from the same Transformer blocks; the difference is which blocks they keep.

ShapeExamplesNaturally good atOur task?
Encoder-only
understands text		 BERT, DistilBERT, RoBERTa, DeBERTa Reading & labeling: classification, sentiment, named-entity recognition (NER), extracting spans. Sees the whole sentence at once (bidirectional). ✓ Best fit
Encoder–decoder (seq2seq)
transforms text		 T5, BART, FLAN-T5 Turning one text into another: translation, summarization, "rewrite as…". Reads an input, writes an output. ~ Works, heavier
Decoder-only
generates text		 GPT, Llama, Mistral, Gemma, Qwen Open-ended generation, chat, reasoning, code. Predicts the next token, left to right. ~ Overkill here

Our job — "find the phone numbers in this note" — is fundamentally a labeling problem: for each word, is it part of a phone number or not? That is exactly what encoder-only models with a token-classification head do best, with the least data and compute. So we'll fine-tune DistilBERT, a small, fast, distilled version of BERT.

Why not just use ChatGPT / a big LLM? You could prompt a large decoder model to extract phone numbers, and it would mostly work. But a fine-tuned DistilBERT is ~100 MB, runs on a CPU in milliseconds, costs nothing per call, needs no internet, and is more consistent on your formats. For a narrow, high-volume task, small-and-fine-tuned beats big-and-general.

6. What actually changes when you fine-tune

You asked the right questions: do we adjust existing layers, or add new ones? Both happen, and it's worth seeing them clearly.

Adding a "head"

A pre-trained encoder like DistilBERT outputs a rich numeric vector for every token. To do NER you bolt a small new layer on top — the head — that maps each token's vector to your label set (here: O, B-PHONE, I-PHONE). That head starts out random. So fine-tuning here does both: it adds a new output layer and adjusts the existing body so its representations suit your task. Hugging Face does this for you with one line: AutoModelForTokenClassification.from_pretrained(...).

Full fine-tuning vs. parameter-efficient (LoRA / PEFT)

ApproachWhat it updatesCostWhen
Full fine-tuningEvery weight in the model.Highest memory; best for small models like DistilBERT.Small models, or you have the GPU.
LoRA / PEFTA tiny set of added adapter weights; the original model is frozen.Tiny — fine-tune a 7B model on one consumer GPU.Large decoder models; limited hardware.

Our example uses full fine-tuning because DistilBERT is small enough that it's simplest and works great on a laptop. For large LLMs you'd typically reach for LoRA — same idea, you just freeze the body and train small adapters bolted alongside it.

7. Tangible steps: finding a fine-tunable model on Hugging Face

Go to the Models hub

Open huggingface.co/models. The left sidebar is your filter panel.

Filter by task

Under Tasks, pick the shape of your problem. For us that's Token Classification (NER lives here). For "rewrite/summarize" you'd pick Summarization; for chat, Text Generation.

Read the model card

Click a model (e.g. distilbert-base-uncased). The card tells you the architecture, the license, intended use, and often example code. A downloadable "Files and versions" tab with a .safetensors / .bin file = open weights = fine-tunable.

Confirm it's a base, not a closed API

If you can download the weights and the license permits training, you can fine-tune it locally. If a "model" is only reachable through a paid endpoint with no weights file, it's closed — see §4.

See what others fine-tuned it into

On a base model's page, the "Adapters / Finetunes" links and a hub search like dslim/bert-base-NER show real fine-tuned descendants — proof the base is trainable and a starting point you can often reuse.

8. Data — the genuinely hard part

The truth nobody tells beginners Writing the training code is the easy 10%. Getting good, correctly-labeled data is the other 90% — and it is what determines whether your model works. The model can only learn the patterns your labels actually contain.

How much do you need?

For a narrow extraction task like ours, a few hundred to a couple thousand labeled examples can already work well, because we start from a pre-trained model. The number grows with task difficulty and the variety of inputs you must handle.

What "good data" means

  • Representative: it must contain every messy variation you'll see in production — (123) 456-7890, 123-456-7890, 1234567890, 123.456.7890, spaces, a leading +1, even 7-digit local numbers.
  • Includes hard negatives: the things that look like phone numbers but aren't — account numbers, order IDs, dates, dollar amounts, ZIP codes. Without these, the model learns "long run of digits = phone" and fails exactly where a regex already fails.
  • Consistently labeled: every annotator must mark spans the same way. Ambiguity in labels becomes confusion in the model.
  • Cleanly split: divide into train / validation / test with no overlap. You tune on validation and report honesty on test. Leakage between splits gives you fake-good numbers.

Where data comes from

  • Your own historical records — here, real (anonymized!) agent notes. The gold standard, because it matches production exactly.
  • Human labeling — annotators mark the spans, often in a tool like Label Studio or Doccano, producing character offsets per entity.
  • Programmatic / synthetic generation — when real labeled data is scarce, generate it: drop known phone numbers (and known distractors) into realistic templates, and you get perfect labels for free. This is exactly what our example does to get you running today, and it's a legitimate way to bootstrap before you mix in real data.
The label format we use (BIO tagging) NER labels each token: B-PHONE = first token of a phone number, I-PHONE = a continuing token of one, O = everything else ("Outside"). So "call me at (123) 456-7890 thanks" tags (123)→B-PHONE, 456-7890→I-PHONE, and every other word →O. This is the standard, model-friendly way to mark spans.

9. Our scenario

We have a pile of free-text notes written by customer-service agents. They contain phone numbers in wildly inconsistent formats, mixed in with account numbers, order IDs, dates, and dollar amounts. We want a model that cleanly pulls out only the phone numbers.

Yes — a regex can catch many phone formats. But a regex also happily grabs a 10-digit account number, and tuning a regex to know "this digit-run is an account, that one is a phone" from surrounding words gets ugly fast. A fine-tuned model learns from context ("call back at…", "acct #…") which digit-runs are phones — the thing regex can't do.

A practical hybrid In production, many teams combine both: the model finds candidate spans using context, then a regex/normalizer cleans the digits into a canonical +1XXXXXXXXXX. Use each tool for what it's good at.

10. The worked example (run it yourself)

The companion code in this repo fine-tunes distilbert-base-uncased into a phone-number extractor. Everything is managed with uv — no manual virtualenvs, no pip install juggling.

The three steps

# 0. one-time: install uv (https://docs.astral.sh/uv/)
#    macOS/Linux:  curl -LsSf https://astral.sh/uv/install.sh | sh

# 1. generate a labeled, synthetic dataset (train/val/test JSONL)
uv run generate-data

# 2. fine-tune DistilBERT into a token-classifier (writes ./phone-ner-model)
uv run train

# 3. try it on new notes
uv run predict "Cust called from (415) 555-0132, acct 5567891234, owes $42.50"

You don't run pip install at all — uv run reads pyproject.toml, builds the environment on first call, and caches it. The first run downloads PyTorch + the base model (a few hundred MB); after that it's instant.

What each file does

FileRole
src/phone_ft/generate_data.pyBuilds realistic notes with phone numbers in every format plus hard-negative distractors, and emits perfectly-labeled BIO data. This is the "get the data" step, automated.
src/phone_ft/train.pyLoads the data, tokenizes it, aligns labels to sub-word tokens, and fine-tunes DistilBERT with the Hugging Face Trainer. Reports precision/recall/F1.
src/phone_ft/predict.pyLoads your fine-tuned model and extracts phone numbers from any new note you pass in.

The heart of it (token-classification fine-tune)

The conceptual core, in a few lines — the full, runnable version is in train.py:

from transformers import (AutoTokenizer, AutoModelForTokenClassification,
                          Trainer, TrainingArguments, DataCollatorForTokenClassification)

labels = ["O", "B-PHONE", "I-PHONE"]
tok = AutoTokenizer.from_pretrained("distilbert-base-uncased")

# Adds a fresh classification HEAD on top of the pretrained body:
model = AutoModelForTokenClassification.from_pretrained(
    "distilbert-base-uncased", num_labels=len(labels))

11. Quick checklist & common pitfalls

  • Can I even fine-tune this model? Are the weights downloadable and the license permissive? If not, it's closed — wrap it instead (§4).
  • Right architecture for the task? Labeling → encoder; transform text → seq2seq; generate → decoder (§5).
  • Do I really need fine-tuning? Try prompting / RAG first; fine-tune for behavior & form (§2).
  • Is my data representative and full of hard negatives? This is where models live or die (§8).
  • Clean train/val/test split, no leakage? Otherwise your metrics lie.
  • Pitfall — too few epochs / too many: too few and it underfits; too many and it memorizes (overfits). Watch validation F1, not training loss.
  • Pitfall — label misalignment: with sub-word tokenizers you must align each label to the right sub-token (the code shows how).
